    Week 5 (Wednesday): Microcontroller Connectivity & Communication Protocols

    Objective

    To understand the fundamental methods by which microcontrollers talk to each other and external peripherals. This session covered general hardware concepts and took a deep dive into serial communication protocols (SPI, UART, I2C) and Wi-Fi.

    General Hardware & System Notes

    Before diving into the protocols, we covered a few foundational microcontroller concepts:

    • Built-in LED: On standard Arduino boards (like the UNO), the built-in onboard LED is hardwired to Digital Pin 13.
    • Watchdog Timer: A hardware fail-safe. The CPU must continuously reset this timer during normal operation; if the system hangs or gets stuck in an infinite loop and fails to reset it, the watchdog forcibly reboots the microcontroller.
    • Shift Registers: Hardware chips used to convert serial data streams into parallel data outputs (or vice versa), which is a common way to expand the number of I/O pins available on a microcontroller.
    • Bit Banging: A technique where communication protocols are implemented entirely in software by manually toggling GPIO pins HIGH and LOW in the code, rather than relying on dedicated hardware circuits inside the chip.

    Communication Theory

    Electronic devices communicate by sending bits of data through physical wires connected between them.

    • Serial vs. Parallel: Serial communication sends data bit-by-bit over a single wire, while parallel communication sends multiple bits simultaneously over multiple parallel wires.
    • Baud Rate: This is a measure of the speed of data transfer, expressed in bits per second (bps).
    • Synchronous Communication: This relies on a shared clock signal to coordinate data transfer. The clock signal is typically generated by the master device and used by both the sender and receiver to ensure data is read or written at the exact same time.
    • Asynchronous Communication: This does not use a shared clock signal. Instead, each data transmission includes special bits, such as start and stop bits, to indicate the beginning and end of data packets, allowing the receiver to determine the timing of each byte independently.

    The Core Protocols

    1. SPI (Serial Peripheral Interface)

    SPI is a synchronous communication protocol commonly used by SD card readers, RFID modules, and wireless transceivers.

    The Hardware Lines:

    • MOSI (Master Output/Slave Input): The line for the master to send data to the slave. Data is usually sent with the most significant bit first.
    • MISO (Master Input/Slave Output): The line for the slave to send data back to the master. Data is usually sent with the least significant bit first.
    • SCLK (Clock): The line for the clock signal that synchronizes the data bits.
    • SS/CS (Slave Select/Chip Select): The line for the master to select which specific slave to send data to. The master activates a slave by setting its SS/CS line to a low voltage level.

    Architecture Note: When using multiple slaves, the MOSI, MISO, and SCLK lines are shared, but the Chip Selects can be connected in parallel (requiring a dedicated SS pin on the master for every slave) or daisy-chained.

    2. UART (Universal Asynchronous Receiver-Transmitter)

    UART is not a protocol, but rather a physical circuit inside a microcontroller (or a standalone IC) whose main purpose is to transmit and receive serial data asynchronously.

    How it Works:

    • It uses two cross-connected lines: Tx (Transmit) and Rx (Receive).
    • Data is transferred from the device's data bus to the transmitting UART in parallel form.
    • The transmitting UART adds a start bit, an optional parity bit, and 1 to 2 stop bits to create a serial data packet (or data frame).
    • It outputs this packet serially, bit by bit, out of the Tx pin.
    • The receiving UART reads the packet at its Rx pin, removes the start/parity/stop bits, and transfers the raw data back into parallel form for the receiving device.

    3. I2C (Inter-Integrated Circuit)

    I2C is a synchronous, serial communication protocol where data is transferred bit by bit along a single data wire. (Application Note: When you tilt your phone, the accelerometer/gyroscope sensor inside is almost certainly communicating with the main processor using I2C!)

    The Hardware Lines:

    • SDA (Serial Data): The single line used for the master and slave to both send and receive data.
    • SCL (Serial Clock): The line that carries the clock signal, which is always controlled by the master.

    How it Works (Master-Slave Architecture):

    • Addressing: Because I2C doesn't have individual Slave Select lines like SPI, it uses an address frame to identify the target slave. Each I2C device has a unique 7-bit or 10-bit address, allowing up to 128 or 1,024 unique devices to share the exact same SDA and SCL lines.
    • Bus Setup: To connect multiple slaves to a single master, 4.7K Ohm pull-up resistors must connect both the SDA and SCL lines to Vcc.
    • Message Framing: A Start Condition occurs when the SDA line drops low before the SCL line drops low. A Stop Condition occurs when the SDA line goes high after the SCL line goes high.
    • ACK/NACK: Each frame in a message is followed by an acknowledge or no-acknowledge bit. For example, during a single byte read, if a data frame is successfully received, the receiving device returns an ACK bit to the sender.

    4. Wi-Fi

    Wi-Fi enables devices to connect to external networks as clients or host networks as servers.

    • It operates on standard 802.11 protocols across 2.4 GHz and 5 GHz frequencies.
    • In the context of IoT and microcontrollers, Wi-Fi communication often involves higher-level protocols like HTTP or MQTT, providing standard ways to send and receive data over the internet.
